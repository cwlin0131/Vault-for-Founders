# Voice & Tone Template: Writing Rules for Public Content

> Last updated: 2026/04/17

---

## Instructions for the AI Setting This Up

**You're the AI helping the founder build this file. Don't fill it in yourself. Discuss first.**

This file captures the founder's **personal voice preferences and writing pet peeves** for public-facing content. Other AIs will consult this file when drafting LinkedIn posts, landing pages, pitch decks, README files, press quotes, etc. The rules must come from the founder, not from a generic best-practices list.

**Suggested discussion flow:**

1. Ask first: "What writing patterns instantly scream 'this was written by AI' to you?"
2. Capture the specifics they name. Those become **Rule #1** (what they notice most).
3. Ask: "What should your public writing feel like? Punchy? Thoughtful? Emotional? Buttoned-up?"
4. Offer relevant items from the "Common Rules Menu" below for them to pick.
5. Ask: "Are there specific words or sentence structures you absolutely refuse to use?" That becomes a banned list.

Write rules into the file only after discussion. **Don't stuff the file with generic best practices. Only include rules the founder will actually enforce.**

---

## What This Is

A set of writing rules that all AIs should read before drafting public content for the founder. The goal: prevent AI-generated content that doesn't sound like the founder.

## Scope

**Apply to:** LinkedIn posts, Twitter / X, Threads, GitHub READMEs, **GitHub Release notes**, landing page copy, blogs, email campaigns, press releases, pitch decks, speaking engagement bios, in-app product copy, public issue / PR descriptions.

**Judgment rule**: if the text will be seen by anyone other than the founder, it counts as public. When unsure, treat it as public.

**Don't apply to:** Internal docs, Linear / Jira issues (internal), Slack DMs, Notion notes, memos, agent-to-agent coordination files.

## Common Rules Menu (for discussion inspiration)

Rules other founders have adopted. Treat as a starting point. **Don't copy wholesale.** Pick 3 to 5 that actually match the founder's instincts.

### 🔤 Punctuation / Symbols

- **No em-dashes (`—`)**: One of the most recognizable AI-writing tells. Readers immediately think "this was written by a bot." Use periods to split, colons to elaborate, or commas to link instead.
  - `Hyphen -` (compounds like `AI-native`) is fine
  - `En-dash –` (ranges like `2022–2026`) is fine
  - Only ban em-dashes
- **No exclamation marks** (or at most one per post)
- **No ellipses `...`**

### 📏 Sentence Structure

- **No "not just X, but Y" constructs**. AI's favorite contrast formula.
- **No "It's not a tool, it's a partner"-style reveals**
- **No parallel triplets** ("Faster, smarter, stronger.")
- **No paragraphs longer than 3 sentences**. Long paragraphs read machine-generated.

### 📣 Word Choice / Tone

- **No empty adjectives**: world-class, revolutionary, cutting-edge, next-generation, game-changing
- **No "leverage," "empower," "unlock,"** or similar corporate buzzwords
- **No "We believe" / "We're on a mission to"** openings
- **No false modesty**: "just a quick thought," "my humble take"

### 🎭 Voice / Perspective

- **No royal "we"** when posting personally
- **No third-person self-reference** (unless it's a press bio)
- **No ghostwriting as "the team"** when it's actually you

## How Other AIs Should Use This File

1. Before drafting any public-facing content, **read this file first**.
2. If a draft breaks a rule, **fix it directly**. Don't ask permission each time.
3. If the founder explicitly says "I want this em-dash here," respect that one-off decision.
4. Add new rules by appending a new `## Rule #N: Name` section at the end.
